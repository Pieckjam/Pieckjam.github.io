Hexo + Butterfly 博客维护手册（命令 / 原理 / 经验 / 排障）

发表于2026-03-27|更新于2026-03-27|Blog
|总字数:2.9k|阅读时长:10分钟|浏览量:

适用范围：本仓库为 Hexo 7.3.0 + Butterfly 主题（theme: butterfly），部署到 Pieckjam.github.io（GitHub Pages）。

使用方式：把它当“速查 + 说明书”。每次维护/更新/迁移/排障时，从对应章节直接抄命令或按清单检查。

0. 先讲清楚：Hexo 在做什么？（整体原理）

Hexo 是一个静态站点生成器。它的核心流程可以用一句话概括：

  1. 你在 source/ 里写内容（Markdown、页面、图片等）
  2. Hexo 读取配置（_config.yml + 主题配置等）
  3. Hexo 把 Markdown 渲染成 HTML，并把主题模板、CSS/JS、资源一起“编译”到 public/
  4. 部署：把 public/ 里生成的静态文件推送到 GitHub Pages（或任意静态托管）

为什么 GitHub Pages 能跑？

GitHub Pages 本质上是：帮你把一个分支（或某个目录）里的静态文件通过 CDN 对外提供。

  • 你的仓库里写的是“源码”（source/、主题、配置等）
  • hexo g 生成的是“产物”（public/ 里的纯静态文件）
  • hexo d 做的是“把产物推到 Pages 指定分支/目录”

你当前的部署配置在 _config.yml：

  • deploy.type: git
  • deploy.repo: git@github.com:Pieckjam/Pieckjam.github.io.git
  • deploy.branch: main

也就是说：执行 hexo d 会把 public/ 生成物推送到 Pieckjam/Pieckjam.github.io 的 main 分支。

提醒：你仓库根目录中已经存在 public/，这通常意味着你本地曾经生成过静态文件并可能把它提交进了源码仓库。常见做法是：源码仓库不提交 public/（用 .gitignore 忽略），部署仓库/分支才存 public/。如果你现在这种结构能正常工作，也可以继续沿用，但后面“最佳实践”会解释更推荐的拆分方式。

1. 快速上手：日常维护的 4 个命令（clean / g / s / d）

下面四个命令基本覆盖了 90% 的日常需求。

1.1 hexo clean：清理缓存与旧生成物

用途：

  • 清理 public/（旧的生成结果）
  • 清理缓存（不同版本/主题/插件更新后，缓存可能导致“改了但不生效”）

什么时候用：

  • 修改主题样式/布局、升级依赖后页面不更新
  • 本地运行正常但部署后资源路径不对
  • 生成出现奇怪的残留页面（例如删了文章但归档还在）

常用组合：

  • hexo clean && hexo g
  • hexo clean && hexo g && hexo s

1.2 hexo generate（简写 hexo g）：生成静态文件

用途：

  • 把 source/ + 配置 + 主题渲染成为 public/

输出：

  • 你会看到 public/ 目录更新（HTML/CSS/JS、归档页、标签页等）

常用参数：

  • hexo g --watch：监视文件变化，自动重新生成（适合边写边预览）
  • hexo g --debug：输出更详细的日志（排障时很有用）

1.3 hexo server（简写 hexo s）：本地预览

用途：

  • 在本地启动一个静态服务器预览站点

常见用法：

  • 仅预览：hexo s
  • 指定端口：hexo s -p 4001
  • 监听生成：hexo s（默认就会自动检测并重新生成）

注意：

  • 本地预览看到的是生成后的效果，等价于你部署后别人看到的效果（除了缓存/CDN 等因素）。

1.4 hexo deploy（简写 hexo d）：部署

用途：

  • 把 public/ 里的生成物推送到配置好的远端仓库/分支

你当前使用的是 hexo-deployer-git。

最常见组合：

  • hexo clean && hexo g && hexo d

提示：

  • 如果你想“一键生成并部署”，也可以用：hexo d -g（deploy 前自动 generate）。

2. 用 npm scripts 统一命令（推荐）

你的 package.json 已经提供常用脚本：

  • npm run clean → hexo clean
  • npm run build → hexo generate
  • npm run server → hexo server
  • npm run deploy → hexo deploy

推荐原因：

  • 更一致（团队/未来迁移时更稳）
  • 可以把复杂参数固化进去（比如端口、debug、环境变量）

你可以把日常流程固化为：

  1. 本地预览：npm run server
  2. 部署发布：npm run clean && npm run build && npm run deploy

3. 写作与内容管理：文章 / 页面 / 草稿

3.1 新建文章：hexo new

  • 新建一篇文章：hexo new "我的新文章"
  • 新建草稿：hexo new draft "未完成的文章"
  • 新建页面：hexo new page "about"

生成位置：

  • 文章默认在 source/_posts/
  • 页面默认在 source/<pagename>/index.md（与主题布局相关）

3.2 Front-matter（文章头部）怎么写？

你仓库的 scaffold（模板）在 scaffolds/post.md，当前默认字段是：

  • title
  • date
  • tags

推荐你在实际写作中补充：

  • categories：分类（可以用于导航/归档更清晰）
  • description：摘要（首页卡片更好看，也利于 SEO）
  • keywords：关键字（可选）

示例：

3.3 草稿与发布

  • 默认 Hexo 不会渲染草稿（你 _config.yml 的 render_drafts: false）
  • 本地预览草稿：hexo s --draft 或 hexo g --draft

将草稿发布为正式文章：

  • hexo publish draft "未完成的文章"

小建议：草稿命名时先用“工作标题”，发布前再改为更适合 URL 的正式标题。

4. 资源与图片管理（避免链接混乱）

你当前配置：post_asset_folder: false，意味着：

  • 默认不会为每篇文章创建同名资源文件夹

常见的两种图片管理策略：

策略 A：全站统一资源目录（适合入门）

  • 把图片放到 source/img/ 或 source/images/
  • 文章里用绝对路径引用：/img/xxx.png

优点：简单、直观。
缺点：图片多了后不易按文章归档。

策略 B：开启文章资源文件夹（适合长期维护）

把 _config.yml 改为：

  • post_asset_folder: true

然后每篇文章都会有：

  • source/_posts/xxx.md
  • source/_posts/xxx/（存图、附件等）

优点：文章与资源“打包”，迁移/删除更干净。
缺点：对初学者略复杂，主题/插件有时需配合。

由于你现在站点已经在用 /img/... 的主题资源（比如 themes/butterfly/source/img 以及根目录 public/img），短期建议先用策略 A；等文章多了再切换策略 B。

5. 配置文件：该改哪个？（最容易搞混的一章）

5.1 Hexo 主配置：_config.yml

作用：控制整个站点层面的行为，例如：

  • 站点信息：title/subtitle/description/author/language/timezone
  • 站点 URL：url、permalink
  • 写作与渲染：render_drafts、代码高亮等
  • 主题：theme: butterfly
  • 部署：deploy（你目前用 git 部署到 main）

关键提醒：

  • _config.yml 的 url 现在还是 http://example.com。部署到 GitHub Pages 后建议改成真实域名，否则可能影响 SEO、sitemap、RSS、某些资源绝对路径生成。

例如（根据你的仓库名推断）：

  • url: https://pieckjam.github.io

如果你后续绑定了自定义域名（如 https://blog.xxx.com），也应该改成那个。

5.2 Butterfly 主题配置：_config.butterfly.yml

作用：控制主题样式与布局，例如：

  • 导航菜单 menu
  • 头像、favicon、背景图
  • 首页布局、文章卡片展示
  • TOC、版权信息、代码块样式
  • 搜索、评论、统计、PWA、CDN 等（通常都在这里开关）

经验：

  • 不要直接改 themes/butterfly/_config.yml（如果有），尽量用根目录的 _config.butterfly.yml 覆盖。
  • 这样升级主题时冲突更少。

5.3 主题源码：themes/butterfly/

里面通常包含：

  • layout/：pug 模板（决定 HTML 结构）
  • source/：stylus/css/js/img（主题静态资源）
  • scripts/：Hexo 扩展脚本（在生成时参与处理）

建议：

  • 主题自定义尽量走“配置项”→ “注入额外 CSS/JS” → 最后才改主题源码。
  • 若必须改主题源码，建议 fork 主题或记录补丁，方便以后升级时合并。

6. 搜索、归档、标签这些页面是怎么来的？

Hexo 通过“生成器插件”生成这些页面。你当前依赖里有：

  • hexo-generator-index：首页列表
  • hexo-generator-archive：归档页（/archives/）
  • hexo-generator-tag：标签页（/tags/）
  • hexo-generator-category：分类页（/categories/）
  • hexo-generator-search：生成 search.xml（本地搜索/第三方搜索用）

生成物你可以在 public/ 里看到，例如：

  • public/archives/index.html
  • public/tags/index.html
  • public/search.xml

7. 部署到 GitHub Pages：从命令到发生了什么

当你执行：

  • hexo clean && hexo g && hexo d

实际发生的是：

  1. 清理旧产物与缓存
  2. 重新生成 public/
  3. hexo-deployer-git 会在一个临时目录初始化 git 仓库，把 public/ 当作仓库内容
  4. 把它推送到 _config.yml 里配置的 repo/branch
  5. GitHub Pages 检测到目标分支更新，刷新站点内容

7.1 SSH 权限 vs HTTPS

你使用的是：

  • git@github.com:Pieckjam/Pieckjam.github.io.git（SSH）

这要求本机：

  • 已配置 SSH key
  • key 已添加到 GitHub
  • 本机 git 能正常 git pull/push

如果你遇到部署报错（权限问题），优先检查：

  • ssh -T git@github.com 是否通过
  • 远端仓库权限是否正确

8. 常见问题排查（建议按顺序）

8.1 改了内容但页面不变

检查顺序：

  1. 是否运行了 hexo clean（尤其改主题/插件后）
  2. 是否在错误的环境/目录执行命令（确保在博客根目录）
  3. 本地是否确实生成了新的 public/（看文件修改时间）
  4. 如果是 GitHub Pages：是否部署成功（查看 hexo d 输出、GitHub 仓库提交记录）
  5. 浏览器缓存/Service Worker（如果你开启了 PWA）

8.2 页面样式错乱 / 资源 404

常见原因：

  • _config.yml 的 url、root（如果设置过）不正确
  • 开了 CDN/压缩后路径出错
  • 主题升级导致配置项变更

建议动作：

  • hexo clean && hexo g && hexo s 本地先确认
  • 直接打开浏览器 DevTools → Network 看 404 的资源路径

8.3 hexo d 失败

常见原因：

  • SSH key 未配置/无权限
  • repo/branch 配错
  • GitHub Pages 的 Source 设置与实际分支不一致

建议动作：

  • 检查 _config.yml 的 deploy 段
  • 进入部署仓库看是否收到新提交

9. 最佳实践（长期维护会很爽）

9.1 建议把“源码仓库”和“部署产物”分开

推荐结构：

  • 仓库 A：blog-source（存 Hexo 源码：source/、主题、配置、package.json）
  • 仓库 B：username.github.io（只存生成后的静态文件，也就是 public/ 的内容）

好处：

  • 源码更干净，避免误提交大量生成物
  • 回滚更清晰
  • 部署仓库 commit 历史更“像发布记录”

你当前部署已经是推送到 Pieckjam.github.io，如果你把它当部署仓库，就更符合这个实践。

9.2 升级依赖/主题前先做备份与对比

建议流程：

  1. 提交当前修改（保证 git 干净）
  2. 升级依赖（例如 npm update 或指定包升级）
  3. hexo clean && hexo g && hexo s 本地自测
  4. 再部署

9.3 文章命名与链接稳定

你的 permalink 模式是：

  • :year/:month/:day/:title/

建议：

  • 标题里避免特殊字符
  • 发布后尽量不要修改 title（会影响 URL）
  • 如果必须改，考虑做重定向（需要额外插件/在托管侧处理）

9.4 写作时的“可维护模板”（推荐固定结构）

建议每篇技术文章固定：

  • 背景/动机
  • 结论（先给答案）
  • 步骤/代码
  • 常见坑
  • 参考链接

长期会显著减少你回头维护的时间。

10. 备忘清单（抄它就行）

写一篇新文章并预览

  • 新建：hexo new "标题"
  • 本地预览：hexo s

发布上线

  • hexo clean && hexo g && hexo d

改主题/插件后不生效

  • hexo clean && hexo g && hexo s --debug
