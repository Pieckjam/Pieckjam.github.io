云计算实验：基于 Hexo + Butterfly + GitHub Pages 部署个人博客（环境构建/部署/维护全记录）

发表于2026-03-27|更新于2026-03-27|实验报告
|总字数:2.1k|阅读时长:7分钟|浏览量:

一、实验概述

1. 实验名称

云计算实验：基于云计算环境的网站部署实践——搭建并发布个人博客（Hexo + Butterfly + GitHub Pages）

2. 实验目的

本实验旨在通过实践操作，掌握基于云计算环境的网站部署技能，具体包括：

  1. 构建并验证静态网站生成与本地预览环境；
  2. 实现个人博客/网站的搭建与主题配置；
  3. 掌握静态站点的构建、生成、部署流程；
  4. 撰写实验报告，并将实验报告作为一篇博文发布到自己的博客/网站。

3. 实验原理（为什么这算“云计算部署”）

本实验选择 GitHub Pages 作为云端托管平台，它提供：

  • 基于云端的静态网站托管（自动分发、CDN 加速、HTTPS 支持）；
  • 与 Git 版本控制自然集成：提交即发布；
  • 对静态站点生成器非常友好（Hexo/Hugo/Jekyll 等）。

因此，本实验的技术路线是：

  • 本地：用 Hexo 将 Markdown 内容渲染为静态 HTML/CSS/JS；
  • 云端：把生成的静态文件推送到 GitHub Pages 指定仓库/分支，由云端完成托管与提供访问。

4. 实验环境

说明：下面按“可复现性”写法给出环境。你可以按自己实际情况补充版本号截图。

  • 操作系统：Windows
  • Node.js：建议 18.x 或 20.x（LTS）
  • Git：用于版本管理与部署
  • Hexo：本仓库 package.json 声明 Hexo 7.3.0
  • 主题：Butterfly（已存在于 themes/butterfly/）
  • 云端平台：GitHub Pages（username.github.io）

5. 实验成果

  • 成功搭建 Hexo 博客并启用 Butterfly 主题；
  • 本地可预览、可生成静态文件；
  • 成功部署到 GitHub Pages，实现公网访问；
  • 将本实验报告发布为博客文章。

二、整体架构与工作流程

1. 静态站点生成器的工作方式

Hexo 是静态站点生成器。核心流程：

  1. 内容源（Markdown）：位于 source/；
  2. 配置：位于 _config.yml、_config.butterfly.yml 等；
  3. 主题渲染：位于 themes/butterfly/（模板 + 样式 + 脚本）；
  4. 生成输出：执行 hexo generate 后产物位于 public/；
  5. 部署发布：执行 hexo deploy 推送产物到 GitHub Pages。

可以把它理解为：

  • source/ = 源代码
  • public/ = 编译后的静态文件（可直接上线）

2. GitHub Pages 的托管原理

GitHub Pages 会把你指定的：

  • 某个仓库（如 Pieckjam.github.io）
  • 某个分支（如 main）

上的静态文件，通过云端基础设施对外提供访问。

本仓库在 _config.yml 中配置了 git 部署：

  • deploy.type: git
  • deploy.repo: git@github.com:Pieckjam/Pieckjam.github.io.git
  • deploy.branch: main

含义：执行 hexo d 会把 public/ 的内容推送到上述仓库/分支，从而触发云端站点更新。

三、实验步骤（完整可复现）

1. 准备工作：安装 Node.js 与 Git

  1. 安装 Node.js（LTS 版本），安装后在终端验证：
  • node -v
  • npm -v
  1. 安装 Git，验证：
  • git --version

可选：若部署使用 SSH 地址（git@github.com:...），需要配置 SSH Key。

2. 创建/准备 Hexo 博客项目

本实验直接使用现有 Hexo 项目结构（你当前仓库已包含）：

  • _config.yml（站点主配置）
  • _config.butterfly.yml（Butterfly 主题配置）
  • source/（文章与页面）
  • themes/butterfly/（主题）
  • package.json（依赖与 Hexo 版本）

若从零开始，一般流程是：

  1. npm install -g hexo-cli
  2. hexo init <folder>
  3. npm install

本仓库已经存在 package.json，通常只需要在根目录执行 npm install 安装依赖即可。

3. 安装依赖

在博客根目录执行依赖安装（一次即可）：

  • npm install

安装完成后，Hexo 相关命令即可使用。

4. 配置站点与主题

4.1 配置站点基本信息（_config.yml）

常见需要修改的字段：

  • title：站点标题
  • subtitle / description：站点副标题与描述
  • author：作者
  • language：语言（如 zh-CN）
  • timezone：时区（如 Asia/Shanghai）

4.2 配置站点 URL（强烈建议）

_config.yml 中的 url 会影响：

  • 生成的绝对链接
  • sitemap / RSS
  • 某些主题资源路径

如果你是 GitHub Pages 的用户主页仓库：

  • url: https://<username>.github.io

你当前 _config.yml 里 url 还是 http://example.com，实验报告建议写明应该改成真实地址。

4.3 启用 Butterfly 主题

在 _config.yml 中：

  • theme: butterfly

主题的详细配置在 _config.butterfly.yml，常用功能包括：

  • 导航菜单（menu）
  • 头像、favicon、背景图
  • 代码块样式、目录（TOC）
  • 搜索、评论、统计等

5. 本地预览与生成

5.1 常用命令说明

  • hexo clean：清理缓存与旧生成物
  • hexo g（hexo generate）：生成静态文件到 public/
  • hexo s（hexo server）：启动本地服务器预览
  • hexo d（hexo deploy）：部署到云端（GitHub Pages）

5.2 本地预览（开发验证）

标准流程：

  1. 清理：hexo clean
  2. 生成：hexo g
  3. 预览：hexo s

访问：

  • http://localhost:4000

6. 撰写实验报告并发布为博文

6.1 新建文章

  • hexo new "云计算实验报告"

文章会生成到：

  • source/_posts/

6.2 编写 Markdown 内容

实验报告建议具备以下结构（便于评分）：

  • 实验目的
  • 实验环境
  • 实验原理
  • 实验步骤
  • 实验结果（截图/生成物说明）
  • 问题与解决
  • 实验总结

本篇文章即为实验报告正文。

6.3 本地生成并确认页面效果

  • hexo clean && hexo g && hexo s

确认：

  • 首页文章列表显示正常
  • 文章详情页样式正常
  • 标签/归档页面正常

7. 部署到 GitHub Pages（云端发布）

7.1 部署插件

本项目依赖中已包含：

  • hexo-deployer-git

因此可直接使用 hexo d。

7.2 部署命令

常见一键发布：

  • hexo clean && hexo g && hexo d

或：

  • hexo d -g

7.3 部署成功的验证方式

  1. 查看终端输出是否显示 push 成功；
  2. 打开部署仓库（如 Pieckjam.github.io）确认提交记录更新；
  3. 在 GitHub Pages 设置页确认 Source 分支正确；
  4. 访问站点 URL，确认新文章已经上线。

注意：GitHub Pages 更新有时会有几十秒到几分钟延迟。

四、实验结果展示（建议配图/截图）

这一章建议你在实际提交时补充截图，评分会更友好。

可展示内容：

  1. 本地预览成功截图（浏览器打开 localhost:4000）
  2. hexo g 生成 public/ 成功的终端输出
  3. hexo d 部署成功的终端输出
  4. GitHub Pages 仓库提交记录截图
  5. 公网访问成功截图（打开 https://<username>.github.io）

五、常见问题与解决方案（排障记录）

1. 修改内容后页面不更新

处理：

  • 执行 hexo clean 后重新 hexo g。

原因：

  • 旧生成物/缓存导致页面残留。

2. 部署失败：权限问题（SSH）

现象：

  • hexo d 提示权限不足/无法访问仓库。

解决：

  • 配置 SSH Key，并在 GitHub 添加公钥；
  • 验证：ssh -T git@github.com。

3. 样式错乱或资源 404

排查：

  • _config.yml 的 url 是否为真实域名；
  • 浏览器 Network 中查看 404 的资源路径；
  • 清理后重新生成与部署。

六、维护与更新流程（实验后续可持续）

1. 日常更新文章

  • 新建文章：hexo new "标题"
  • 本地预览：hexo s
  • 发布上线：hexo clean && hexo g && hexo d

2. 推荐用 npm scripts 统一命令

本仓库 package.json 已提供：

  • npm run clean
  • npm run build
  • npm run server
  • npm run deploy

日常可以使用：

  • 本地预览：npm run server
  • 发布上线：npm run clean && npm run build && npm run deploy

七、实验总结与心得

  1. 静态站点生成器把“写作内容”和“站点展示”解耦：内容用 Markdown 写，展示交给主题渲染。
  2. GitHub Pages 提供了云端托管能力，结合 Git 的版本管理，实现了低成本、自动化、可回滚的网站发布流程。
  3. 实践中最常用、最稳妥的一键发布命令是：hexo clean && hexo g && hexo d。
  4. 建议将站点 url 配置为真实地址，并养成“本地验证后再部署”的习惯，可以显著减少线上故障。
